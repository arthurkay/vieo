Executive Summary

vieo is an open-source video streaming platform designed for self-hosted environments. It ingests media from arbitrary sources (files, RTMP streams, RTSP feeds, or capture devices), transcodes them into the HLS (HTTP Live Streaming) format using FFmpeg, and serves the resulting segments over HTTP for playback in any modern web browser via hls.js.

The platform is built around a channel abstraction: each source belongs to a channel, and viewers navigate channels to find live or archived content. A robust job-manager oversees the lifecycle of every transcoding operation — from pending through running to completed or failed — with automatic pause/resume based on disk pressure, and full recovery after service restarts.

Key design goals include:

  • Single-binary deployment — Go produces a statically linked binary with no runtime dependencies beyond FFmpeg.
  • Zero external services — SQLite replaces PostgreSQL/MySQL. No Redis, no message queue.
  • Resilience by default — Jobs survive restarts, disk pressure is handled gracefully, and the system self-recovers.
  • Beautiful UI — A shadcn/ui React frontend that feels modern and responsive.

System Architecture

2.1 Architectural Overview

vieo follows a layered monolith architecture within a single process. All components communicate through Go interfaces, keeping the system testable and maintainable without introducing network overhead between services.

2.2 Data Flow

  1. Ingest — A source is registered via the API (file path, RTMP URL, or device).
  2. Probe — ffprobe extracts codec, resolution, duration, and format metadata.
  3. Transcode — A job is created and dispatched to the worker pool. ffmpeg reads the source and writes HLS segments.
  4. Serve — The HTTP server exposes .m3u8 playlists and .ts segments under /api/stream/{id}/.
  5. Play — The React frontend uses hls.js to fetch and render the stream in a <video> element.
  6. Monitor — WebSocket pushes real-time job progress (status, progress %, logs) to the UI.
Transcoding Pipeline: Source → ffprobe (info) → ffmpeg (decode → encode H.264/AAC) → segmenter (.ts + .m3u8) → HTTP server → hls.js → <video>

Component Design

3.1 Media Engine

The Media Engine (internal/media/) encapsulates all interaction with FFmpeg and ffprobe. It exposes two primary interfaces:

Probe

Wraps ffprobe to extract media metadata. Returns a MediaInfo struct containing:

  • Format: container, duration, bitrate, size
  • Video streams: codec, resolution, frame rate, pixel format
  • Audio streams: codec, sample rate, channels, bitrate

Transcode

Constructs and executes an ffmpeg command that transcodes any input to HLS. The command template: 

ffmpeg -i <input> \
  -codec:v libx264 -preset fast -crf 23 \
  -codec:a aac -b:a 128k \
  -f hls \
  -hls_time 4 \
  -hls_list_size 0 \
  -hls_segment_filename <output_dir>/seg_%05d.ts \
  <output_dir>/playlist.m3u8

Progress is parsed from FFmpeg's stderr (the time=... lines) and pushed into a channel for the Job Manager to consume.

3.2 Job Manager

The Job Manager (internal/job/) owns the lifecycle of every transcoding operation.

State Machine

┌──────────┐ │ pending │ └────┬─────┘ │ start ┌────▼─────┐ ┌────│ running │◄────────────┐ │ └────┬─────┘ │ │ │ fail │ resume │ ┌────▼─────┐ ┌────┴─────┐ │ │ failed │ │ paused │ │ └──────────┘ └────┬─────┘ │ │ disk OK │ ┌──────────┐ │ └────│completed │ │ └──────────┘ │ │ manual stop ┌────▼─────┐ │ stopped │ └──────────┘
StateDescription
pendingJob created, awaiting resources
runningActively transcoding
pausedPaused due to disk pressure or signal
completedSource fully consumed, HLS output ready
failedUnrecoverable error (bad source, corrupt file)
stoppedManually stopped by user

Worker Pool

A configurable number of goroutines (default: 3) consume from a job channel. Each worker manages one ffmpeg subprocess. The pool respects context cancellation for graceful shutdown.

Disk Watcher

A background goroutine ticks every 30 seconds, checking disk usage on the data partition via syscall.Statfs. When usage exceeds VIEO_DISK_WARN (default 90%), running jobs are paused. Below that threshold, paused jobs resume automatically. At VIEO_DISK_CRIT (95%), jobs are force-stopped.

3.3 Database Layer

Uses modernc.org/sqlite — a pure Go SQLite implementation that requires no CGO. Migrations are stored as an ordered slice of SQL strings in internal/db/db.go and applied at startup.

Each model (channel, source, output, job) lives in its own file under internal/db/models/ and exposes CRUD functions that accept context.Context and *sql.DB as first parameters.

3.4 API Layer

Built on go-chi/chi v5, the API server is defined in internal/server/server.go with routes registered in internal/server/routes.go. Handlers are grouped by resource in internal/server/handler/.

Middleware stack:

  • chi.Logger — request logging
  • chi.Recoverer — panic recovery
  • chi.RealIP — trusted proxy IP
  • CORS — for frontend dev server (localhost:5173)

3.5 WebSocket Events

The /api/ws endpoint upgrades HTTP connections to WebSocket. The server pushes typed JSON events to all connected clients:

4.2 Table Definitions

channels

CREATE TABLE channels (
    id         INTEGER PRIMARY KEY AUTOINCREMENT,
    name       TEXT    NOT NULL,
    slug       TEXT    NOT NULL UNIQUE,
    description TEXT   DEFAULT '',
    created_at TEXT    NOT NULL DEFAULT (datetime('now'))
);

sources

CREATE TABLE sources (
    id         INTEGER PRIMARY KEY AUTOINCREMENT,
    channel_id INTEGER NOT NULL REFERENCES channels(id) ON DELETE CASCADE,
    type       TEXT    NOT NULL CHECK(type IN ('file','rtmp','rtsp','device')),
    url        TEXT    NOT NULL,
    metadata   TEXT    DEFAULT '{}',
    created_at TEXT    NOT NULL DEFAULT (datetime('now'))
);

outputs

CREATE TABLE outputs (
    id         INTEGER PRIMARY KEY AUTOINCREMENT,
    source_id  INTEGER NOT NULL REFERENCES sources(id) ON DELETE CASCADE,
    type       TEXT    NOT NULL DEFAULT 'hls',
    path       TEXT    NOT NULL,
    created_at TEXT    NOT NULL DEFAULT (datetime('now'))
);

jobs

CREATE TABLE jobs (
    id         INTEGER PRIMARY KEY AUTOINCREMENT,
    source_id  INTEGER NOT NULL REFERENCES sources(id) ON DELETE CASCADE,
    output_id  INTEGER NOT NULL REFERENCES outputs(id) ON DELETE CASCADE,
    status     TEXT    NOT NULL DEFAULT 'pending'
                      CHECK(status IN ('pending','running','paused','completed','failed','stopped')),
    progress   REAL    NOT NULL DEFAULT 0.0,
    error_msg  TEXT    DEFAULT '',
    pid        INTEGER DEFAULT 0,
    created_at TEXT    NOT NULL DEFAULT (datetime('now')),
    ended_at   TEXT
);

job_logs

CREATE TABLE job_logs (
    id         INTEGER PRIMARY KEY AUTOINCREMENT,
    job_id     INTEGER NOT NULL REFERENCES jobs(id) ON DELETE CASCADE,
    level      TEXT    NOT NULL DEFAULT 'info',
    message    TEXT    NOT NULL,
    created_at TEXT    NOT NULL DEFAULT (datetime('now'))
);

5.3 HLS Streaming

HLS segments and playlists are served as static files from the data directory. The server maps /api/stream/{output_id}/* to the output's path on disk.

Cache headers: Playlists (.m3u8) are served with Cache-Control: no-cache. Segments (.ts) are served with Cache-Control: public, max-age=3600.

6.2 Key Components

VideoPlayer

Wraps hls.js with a clean <video> element. Features: play/pause, seek, volume, fullscreen, picture-in-picture. Supports automatic quality switching via hls.js level selection.

JobStatusBadge

A compact badge component showing job state with color coding:

  • Pending — gray
  • Running — blue + animated pulse
  • Paused — amber
  • Completed — green
  • Failed — red
  • Stopped — slate

ProgressBar

A thin animated progress bar (0–100%) that updates via WebSocket in near real-time. Uses transition: width 0.5s ease for smooth visual updates.

SidebarLayout

The main application shell with:

  • Collapsible sidebar (navigation links, branding)
  • Top header bar (breadcrumbs, system status indicator)

Deployment Guide

7.1 Prerequisites

  • Go 1.22+
  • FFmpeg + ffprobe (available on PATH)
  • Node.js 18+ and npm (for frontend builds)
  • A POSIX-compatible OS (Linux, macOS, BSD)

7.2 Installation

# 1. Clone repository
git clone https://github.com/arthurkay/vieo
cd vieo

# 2. Build backend
go build -o vieo ./cmd/vieo

# 3. Build frontend
cd web
npm install
npm run build
cd ..

# 4. Run
./vieo \
  --db ./data/vieo.db \
  --data-dir ./data/hls \
  --http-addr :8080

The frontend build output is automatically embedded in the Go binary (via embed.FS) and served at the root / when no other route matches.

8.2 Resilience Model

Job Persistence

At startup, the Job Manager queries the database for all jobs left in running or paused state and resumes them. Running jobs from a previous process instance are reset to pending (since their ffmpeg PID is gone), then restarted.

Graceful Shutdown

On SIGTERM / SIGINT:

  1. HTTP server stops accepting new connections.
  2. Running ffmpeg processes receive SIGTERM.
  3. After 5 seconds, any remaining processes receive SIGKILL.
  4. Job states are persisted to the database.
  5. Database connection is closed cleanly.
  6. Process exits.

Process Supervision

If an ffmpeg subprocess exits unexpectedly, the Job Manager marks it as failed and logs the exit code. A configurable retry policy (max_retries: 3 with exponential backoff) can reattempt the job.

8.3 Disk Space Management

ThresholdAction
< 90%Normal operation. All jobs proceed.
90% – 95%Running jobs are paused (SIGSTOP to ffmpeg). Paused jobs resume when disk drops below 85%.
> 95%Running jobs are stopped (SIGTERM → SIGKILL). No new jobs accepted. Admin alert logged.

The disk watcher checks every 30 seconds using syscall.Statfs on the data directory partition.
